## Цель
В таблице `/admin/analysis-bookings` добавить кнопку **«Добавить запись»**, через которую админ создаёт новую запись на анализы для выбранного пациента, задаёт дату, время, адрес и (при необходимости) статус. Защита от дублей: нельзя создать запись для пациента, у которого уже есть **активная** запись.

## UI
- Кнопка **«+ Добавить запись»** в шапке вкладки «Записи» (рядом с фильтрами).
- Открывает диалог `CreateBookingDialog`:
  - **Пациент** — поиск по ФИО/email/телефону (Combobox + запрос к `profiles` с ролью `patient`, дебаунс).
  - **Дата** — Calendar/Popover (минимум сегодня).
  - **Время** — Input `type="time"`.
  - **Адрес** — Input (по умолчанию подставляется адрес из профиля пациента, если есть).
  - **Статус** — Select, по умолчанию `scheduled`; доступны все статусы из текущего `statusLabels` (включая `waiting_call`, `no_answer`).
  - Кнопки: «Отмена», «Создать».
- Если у пациента уже есть активная запись — показываем inline-предупреждение в диалоге («У пациента уже есть активная запись от {дата}. Завершите её или используйте редактирование») и блокируем кнопку «Создать».

## Логика «активной» записи (правило дубля)
Активной считается запись, у которой `status NOT IN ('collected','uploaded')`. То есть всё, что ещё не довезено/не загружено в систему: `not_scheduled`, `scheduled`, `received`, `waiting_call`, `no_answer`.

Повторные анализы (новый цикл раз в 3 мес) не блокируются: предыдущая запись к этому моменту имеет статус `collected`/`uploaded`, и активной для пациента нет.

## Защита от дублей — два уровня

**1. Клиент (UX).** Перед открытием формы и при выборе пациента запрашиваем `analysis_bookings` по `user_id` со статусом NOT IN (`collected`,`uploaded`) — если найдено, показываем предупреждение и блокируем сабмит.

**2. База (надёжно).** Частичный уникальный индекс на `analysis_bookings`:
```sql
CREATE UNIQUE INDEX analysis_bookings_one_active_per_user
ON public.analysis_bookings (user_id)
WHERE status NOT IN ('collected','uploaded');
```
Это гарантирует, что даже при гонке/ручном INSERT нельзя создать вторую активную запись. При срабатывании конструкции — ловим ошибку `23505` и показываем понятное сообщение.

Существующий триггер `create_next_analysis_booking` создаёт следующую запись только когда текущая переходит в `collected` — то есть совместим с индексом (старая уже `collected`, активной нет).

⚠️ **Перед добавлением индекса** нужно убедиться, что в БД сейчас нет пациентов с >1 активной записью. Если есть — в миграции их нужно нормализовать (оставить самую свежую активной, остальные пометить `collected` либо удалить). Этот шаг включаю в миграцию автоматическим запросом, который только проверит и упадёт с понятным сообщением, чтобы вручную решить, что делать (а не молча портить данные).

## Что меняется в коде
- **Новый файл**: `src/components/admin/CreateBookingDialog.tsx` — форма создания.
- **Новый файл**: `src/components/admin/PatientCombobox.tsx` — поиск пациентов (или inline в диалоге).
- **Изменить**: `src/pages/admin/AnalysisBookings.tsx` — кнопка «+ Добавить запись» в шапке вкладки, состояние открытия диалога, invalidate `['analysis-bookings']` после успеха.
- **Миграция**: проверка на существующие дубли + создание частичного уникального индекса.

## Что НЕ делаем
- Не трогаем `availability_slots` (ручная запись админом — вне системы слотов; `slot_id` оставляем NULL).
- Не отправляем email/SMS уведомления при ручном создании (можно добавить позже отдельным переключателем).
- Не трогаем баннер пациента и логику следующего анализа.
