## Проблема

Хук `useDemoMode` используется в 11 разных страницах/компонентах (`Dashboard`, `Biomarkers`, `Analyses`, `MyState`, `HealthStrategy`, `Trends`, `Prescriptions`, `Recommendations`, `AnalysisDetail`, `Profile`, `DashboardLayout`). Каждый вызов хука создаёт **свой собственный локальный стейт** и независимо подгружает `demo_mode_enabled` из БД.

Последствия:
- Когда пользователь нажимает «Выключить демо-режим» в баннере на Контрольной панели, обновляется только локальный стейт этого экземпляра хука. У соседних компонентов (`DashboardLayout`, дочерние карточки, другие страницы) `demoMode` остаётся `true` до полной перезагрузки.
- Поэтому на той же Контрольной панели остаётся второй баннер (от `DashboardLayout`/дочернего компонента) и продолжают рендериться демо-биомаркеры/карточки, потому что они смотрят в свой `demoData`.
- Пользователь ожидает: один переключатель — и демо отключается везде сразу.

## Решение

Превратить `useDemoMode` в глобальный источник истины через React Context.

### Изменения

1. **Новый файл `src/contexts/DemoModeContext.tsx`**
   - `DemoModeProvider` оборачивает приложение и хранит единственный экземпляр стейта: `demoMode`, `demoData`, `loading`.
   - Внутри — текущая логика из `src/hooks/useDemoMode.ts`: `fetchDemoModeStatus`, `loadDemoData`, `adaptDemoDataToUser`, `toggleDemoMode`.
   - Экспортирует хук `useDemoMode()`, который читает контекст; сигнатура возвращаемого объекта (`demoMode, demoData, loading, toggleDemoMode`) сохраняется — все 12 потребителей продолжат работать без правок.

2. **`src/App.tsx` (или корневой layout)**
   - Обернуть дерево в `<DemoModeProvider>` сразу после провайдера аутентификации, чтобы `getUserId()` имел контекст пользователя.

3. **`src/hooks/useDemoMode.ts`**
   - Заменить реализацию на тонкий ре-экспорт `useDemoMode` из нового контекста, чтобы все существующие импорты `@/hooks/useDemoMode` продолжали работать без изменений по всему коду.

4. **Один баннер на странице**
   - Оставить `DemoBanner` только на верхнем уровне страницы (как сейчас в `Dashboard.tsx`, `Biomarkers.tsx` и т. д.).
   - Убедиться, что `DashboardLayout` не рендерит дополнительный `DemoBanner` поверх страничного (если рендерит — перенести его в layout и убрать из страниц, либо наоборот). Финальное правило: **ровно один `DemoBanner`** в дереве на любой странице, и он должен вызывать `toggleDemoMode(false)` из общего контекста.

5. **Сброс данных при выключении**
   - `toggleDemoMode(false)` в провайдере уже делает `setDemoData(null)` и `setDemoMode(false)` — благодаря единому стейту это моментально приведёт все страницы к реальным данным (биомаркеры, карточки, рейтинги систем) без перезагрузки.

### Что НЕ меняем

- Логику адаптации демо-данных под пользователя (`adaptDemoDataToUser`) и шаблон `demoTemplate.json`.
- Поведение `toggleDemoMode(true)` (включение по-прежнему запрещено, если есть реальные анализы).
- Внешний API хука — все 12 потребителей продолжат работать без правок.

## Результат

После применения: один клик «Выключить демо-режим» в баннере на Контрольной панели:
- мгновенно убирает баннер (он у пользователя один),
- обнуляет `demoData` глобально, поэтому биомаркеры/карточки/рейтинги систем тут же переключаются на реальные данные пользователя,
- при переходе на «Мои анализы», «Стратегию здоровья» и т. д. — там тоже больше нет демо.
