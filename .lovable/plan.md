## Цель

Добавить отображение блоков **«Питание и образ жизни»** и **«Дополнительные обследования / консультации»** в админ-интерфейсе. Сейчас данные есть в БД (`recommendations.type='Назначения'` → `content_json.lifestyle` + `content_json.follow_ups`), но видны только нутрицевтики (отдельная таблица `prescriptions`).

## Что изменится

### 1. EditReportDialog (диалог «Редактировать отчёт» в админке)

Файл: `src/components/admin/EditReportDialog.tsx`

- В `loadRecommendations` уже грузятся все строки по `analysis_id`, включая `type='Назначения'`. Но раздел «Назначения» сейчас рендерится через ReactQuill (как HTML-текст), что бессмысленно — он содержит JSON-данные, а не markdown.
- Изменить логику отображения раздела `Назначения`:
  - Если у секции есть `content_json` с полями `lifestyle`/`follow_ups` → рендерить кастомным компонентом (read-only список по группам: питание / активность / сон / специалисты).
  - Под этим списком оставить блок нутрицевтиков (текущий `prescriptions`), как «Нутрицевтики (N)».
- Заменить пункт `«Назначения (N)»` в селекторе разделов: вместо отдельного псевдо-пункта `prescriptions` — единый раздел «Назначения», который показывает 3 секции в одном свитке:
  1. Нутрицевтики (карточки из `prescriptions` с кнопками «Подтвердить» / «Редактировать» — как сейчас)
  2. Питание и образ жизни (3 группы из `content_json.lifestyle`)
  3. Дополнительные консультации (`content_json.follow_ups` — список с указанием специалиста, цели и триггера)
- Не трогать редактирование других markdown-разделов.

### 2. PatientView /prescriptions (View As Patient)

Файл: `src/pages/Prescriptions.tsx`

Страница уже умеет рендерить блоки lifestyle и follow_ups (функция в районе строк 325–400). Нужно убедиться, что:
- Запрос `recommendations-advisory` корректно отрабатывает, когда админ заходит как пациент (`viewAsUserId` есть, `demoMode` отключается на этом пути).
- Лог `[advisory] fetch result` показывает реальную строку. Если нет — проверить RLS-политику для админа с `patients`-permission: она уже добавлена прошлой миграцией, должно работать. При необходимости — снять `eq("type", "Назначения")` фильтр и логировать всё, что вернулось.
- Параметры `queryKey` включают `viewAsUserId`, чтобы кеш не залипал при переключении между пациентами.

### 3. Совместимость данных

Структура `content_json` в БД:
```json
{
  "lifestyle": { "nutrition": [...], "activity": [...], "sleep": [...] },
  "follow_ups": [ { "specialist": "...", "goal": "...", "trigger": "..." } ]
}
```

Тип `FollowUp` в `Prescriptions.tsx` уже соответствует (`specialist/goal/trigger`). В EditReportDialog нужно объявить тот же тип.

## Технические детали

- Никаких миграций БД — данные уже валидны.
- Никаких изменений в edge-функции `analyze-biomarkers` — она уже сохраняет правильный JSON.
- Изменения только во фронтенде: 2 файла (`EditReportDialog.tsx`, `Prescriptions.tsx` — точечная диагностика).
- Read-only визуал в админке для lifestyle/follow_ups (редактирование текста этих пунктов — отдельная задача в будущем, сейчас админ их утверждает вместе со статусом анализа).

## Что НЕ делаем

- Не добавляем UI для редактирования lifestyle/follow_ups (только просмотр + статус).
- Не меняем порядок блоков на странице пациента (нутрицевтики сверху, lifestyle и follow_ups ниже — как просил пользователь ранее).
- Не трогаем PDF-генерацию (она читает те же данные через snapshot).
