## Задача

Разобраться, почему в `EditReportDialog` (админ → Дарбинян) видны только нутрицевтики, а блоки **«Питание и образ жизни»** и **«Дополнительные консультации»** не отображаются, хотя данные в БД есть и валидны.

## Что мы уже знаем

- В БД для Дарбинян есть строка `recommendations` с `type='Назначения'` и валидным `content_json` (3147 байт): `lifestyle.{nutrition,activity,sleep}` + `follow_ups` заполнены.
- Заголовок аккордеона показывает «Назначения (10)» — счётчик берётся только из `prescriptions`, поэтому это **не** доказывает, что advisory загрузилось.
- RLS политики `recommendations` корректные: staff с permission `patients` имеет SELECT-доступ.
- Код рендера в `EditReportDialog.tsx` (строки 438–501) корректный: если `advisory` не null и есть данные — блоки рисуются.

## Гипотезы

1. **Данные не доходят до клиента** — RLS отдаёт строку, но `content_json` приходит как `null` (возможно, из-за устаревшего select в supabase types или особого поведения jsonb).
2. **`setAdvisory` вызывается, но advisory остаётся null** из-за условия пустоты на строке 128.
3. **Запрос вообще не отрабатывает** для админа, использующего «View As» (например, `analysisId` приходит другим, или RLS контекст отличается).

Без рантайм-логов однозначно ответить нельзя — поэтому сначала добавим диагностику и защитные правки, а после прогона прочитаем консоль и поправим точечно.

## Изменения

### Файл: `src/components/admin/EditReportDialog.tsx`

#### 1. Добавить диагностические логи в `loadRecommendations`

После выполнения запроса логируем:
- количество строк
- список `type` всех строк
- найден ли `advisoryRow`
- ключи `content_json` и длины `lifestyle.*` / `follow_ups`

Префикс `[EditReportDialog]` — чтобы быстро найти в консоли через `code--read_console_logs`.

#### 2. Снять «фильтр пустоты» для advisory

Сейчас:
```ts
if (lifestyleCount > 0 || followUps.length > 0) {
  setAdvisory({ lifestyle, followUps });
} else {
  setAdvisory(null);
}
```

Меняем на:
```ts
// Если строка «Назначения» вообще существует — показываем блок,
// даже если массивы пустые (отрисуем placeholder).
setAdvisory({ lifestyle, followUps });
```

Это гарантирует, что админ увидит сам факт раздела, даже если ИИ вернул пустые массивы.

#### 3. Поправить заголовок аккордеона

Сейчас: `Назначения (10)` — только prescriptions.

Меняем на состав:
```
Назначения (нутрицевтики: 10 · образ жизни: N · консультации: M)
```

Если какое-то N/M = 0 — пропускаем соответствующую часть. Это сразу визуально подскажет, видит ли админ advisory.

#### 4. Добавить fallback-секцию «Нет данных» для advisory

Если `advisory` есть, но все массивы пустые — показываем явный текст «AI не сгенерировал блок lifestyle/follow_ups для этого анализа» вместо пустоты, чтобы админ понимал, что проблема на стороне генерации, а не отображения.

### План диагностики после деплоя

1. Открыть в админке EditReportDialog для Дарбинян.
2. Прочитать логи через `code--read_console_logs` с фильтром `EditReportDialog`.
3. По логам сразу будет видно:
   - **Если строк `Назначения` нет в выборке** → проблема в RLS / запросе → копаем дальше.
   - **Если строка есть, но `content_json` = null** → проблема в типах supabase или в `select("*")` → пробуем явный select.
   - **Если данные есть и advisory выставлен** → проблема была в фильтре пустоты (уже исправлено пунктом 2).

## Что НЕ делаем

- Никаких миграций БД.
- Не трогаем edge-функцию `analyze-biomarkers`.
- Не трогаем страницу пациента `/prescriptions`.
- Не добавляем редактирование lifestyle/follow_ups (только просмотр).

## Ожидаемый результат

После прогона мы увидим в консоли точную причину, и уже сейчас:

- Блок advisory не будет «прятаться» при пустых массивах.
- Админ увидит явное сообщение, если AI не вернул lifestyle/follow_ups.
