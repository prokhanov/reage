## Что делаем

В «Просмотре как пациент» → «Новый анализ» добавляем второй способ ввода: загрузить пачку PDF и дать ИИ их распознать.

## UI

### 1. Шаг 1 — две вкладки (Tabs)

Файл: `src/components/admin/AnalysisStep1.tsx` (плюс лёгкая правка `CreateAnalysisWizard.tsx`).

- Вкладка **«Вручную»** — текущая форма (дата + лаборатория + кнопка мок-данных). Поведение не меняется.
- Вкладка **«Авто (PDF)»** — новый компонент `AnalysisAutoImport.tsx`.

Когда юзер на вкладке «Авто», шаги 2–3 мастера скрываются — финальное действие («Импортировать N анализов») происходит на этой же вкладке и закрывает мастер с `onSuccess()`.

### 2. Компонент `AnalysisAutoImport.tsx`

- Drag-and-drop зона + `<input type="file" multiple accept="application/pdf">`. До 10 файлов за раз, ≤20 МБ каждый.
- Список загруженных файлов с статусами: `queued → uploading → parsing → done / error`.
- При нажатии «Распознать»:
  1. Каждый PDF заливается в приватный бакет `analysis-uploads` под путём `{viewAsUserId}/{uuid}.pdf`.
  2. Параллельно (по 2–3 за раз, чтобы не упереться в rate-limit) вызывается edge-функция `parse-analysis-pdf` со ссылкой на файл.
  3. По мере возврата результата под карточкой файла рисуется отчёт распознавания (см. ниже).
- Кнопка «Импортировать выбранное» создаёт `analyses` + `analysis_values` для каждого подтверждённого файла (по одной записи на файл; если в файле обнаружено несколько разных дат сдачи — несколько записей). Использует тот же путь сохранения, что и ручной шаг 3 (без авто-генерации отчёта; чек-бокс «генерировать отчёт после импорта» оставляем выключенным по умолчанию).

### 3. Отчёт по файлу (под каждой карточкой)

Группы:
- **Распознанная шапка**: лаборатория, дата сдачи (с возможностью поправить инлайн).
- **Распознанные показатели** — таблица: показатель (как напечатано) → биомаркер в нашей панели → значение → единица (raw / нормализованная) → страница → бейдж статуса:
  - `ok` — зелёный
  - `unit_mismatch` — жёлтый, с подсказкой ожидаемой единицы и (если возможно) предложением автоконвертации
  - `low_confidence` — жёлтый
  - `value_parse_error` — красный
- **Не сопоставлено с панелью** (`unknown_biomarker`) — отдельный сворачиваемый список «Эти показатели в файле есть, но в нашей панели их нет / синоним не распознан». Печатается имя как в PDF + значение + страница. Эти строки не импортируются.
- Чек-боксы у каждой ok-строки — что именно импортировать. По умолчанию `ok` и `unit_mismatch` (после ручной правки) включены, остальные выключены.

## Бэкенд

### 4. Приватный бакет `analysis-uploads`

- Создаём через `storage_create_bucket` (private).
- RLS на `storage.objects`: insert/select/delete разрешены пользователю на путях `{auth.uid()}/...`; для суперадмина и стаффа с правом `patients` — на любые пути. Файлы пилит EF после обработки (или оставляем TTL-чистильщик отдельно — не в этом плане).

### 5. Edge-функция `parse-analysis-pdf`

- Auth: проверка JWT, требование `has_role(superadmin)` или `has_admin_permission(patients)`.
- Body: `{ patientId, storagePath }`.
- Шаги:
  1. Подписать URL объекта в бакете (`createSignedUrl`, 5 минут) или скачать в EF и собрать `data:application/pdf;base64,...`. Из-за лимита тела EF (~6 МБ) и того, что PDF может быть тяжёлым, используем signed URL и передаём его в `{type:"image_url"}`-формате multimodal? — нет, для PDF в Gateway нужен `{type:"file", file:{file_data:"data:...;base64,..."}}`. Поэтому: скачиваем PDF в EF, кодируем в base64, шлём как `file`-блок.
  2. Грузим каталог биомаркеров из БД: `id, code, name, unit, category`.
  3. Зовём Lovable AI `google/gemini-2.5-pro` (мультимодал; PDF до 50 стр.). System prompt = «ты лабораторный парсер, верни строгий JSON по схеме» + полный каталог биомаркеров (code, name, unit) как справочник для сопоставления.
  4. Запрашиваем структурированный JSON:
     ```
     {
       lab_name, collection_date,                   // как в PDF
       items: [{
         printed_name, value_raw, unit_raw,
         page, ref_range_raw, biomarker_code|null,  // если AI уверен
         confidence: 0..1
       }],
       notes
     }
     ```
  5. Серверная пост-обработка:
     - Сопоставляем `biomarker_code` ↔ каталог. Если код не нашёлся — пробуем match по нормализованному имени (lowercase, без скобок/пунктуации) против `name`. Если и так нет — `status: "unknown_biomarker"`.
     - Парсим `value_raw` (заменяем запятую на точку, убираем `<`/`>` — помечаем `value_parse_error`, если не число).
     - Сравниваем `unit_raw` с `biomarker.unit`. Точное совпадение → `ok`; ожидаемая, но известная конверсия (мг/дл↔ммоль/л для глюкозы/холестерина — табличка в `_shared/unitConvert.ts`) → `unit_mismatch` с предложением `value_converted`. Иначе `unit_mismatch` без конверсии.
     - `confidence < 0.6` → флаг `low_confidence`.
     - Не лезем в `analyses`/`analysis_values` — только возвращаем структуру. Запись в БД делает фронт по подтверждению.
- Ошибки Gateway 429/402 — поднимаем наверх с понятным сообщением (UI показывает в карточке файла).

### 6. Сохранение в БД (фронт)

При нажатии «Импортировать выбранное» — на каждый файл:
- `insert into analyses (user_id, date, lab_name, status='processed')`,
- `insert into analysis_values (analysis_id, biomarker_id, value, unit_override?)` пакетно для выбранных строк (`unit_override` ставим только если юзер оставил единицу, отличную от `biomarker.unit`),
- инвалидация кешей анализов (как в существующем `handleSave`).

## Что НЕ делаем

- Не трогаем ручную ветку и `EditAnalysisWizard`.
- Не автогенерируем отчёт по импортированным анализам — оставляем галку, которой пользуется юзер вручную потом, как сейчас.
- Не делаем OCR для сканов плохого качества отдельно — полагаемся на возможности Gemini 2.5 Pro (он сам распознаёт текст в PDF). Если файл не распознался — статус `error` в карточке.
- Не пишем background-очередь / `parse_jobs` таблицу — обработка идёт в открытом мастере. Если юзер закроет диалог во время парсинга, прогресс теряется.
