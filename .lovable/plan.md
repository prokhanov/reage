## Причина
Сейчас и `src/pages/Index.tsx` (лендинг), и `src/components/ProtectedRoute.tsx` блокируют рендер до тех пор, пока не вернётся `supabase.auth.getSession()`. У этого вызова нет таймаута. Если Lovable Cloud / Auth-сервис перезапускается (как было в твоём случае в 23:53 UTC), либо сессионный токен зависает на refresh — промис никогда не резолвится и пользователь видит чёрный экран со спиннером бесконечно.

В логах подтверждается:
- `received graceful shutdown signal` → перезапуск GoTrue
- Множество `GET /user` запросов с твоего IP (несколько открытых вкладок усугубляют нагрузку)

## Что меняем

### 1. `src/pages/Index.tsx`
- Обернуть `supabase.auth.getSession()` в `Promise.race` с таймаутом **2.5 сек**.
- При таймауте/ошибке — считать пользователя неавторизованным и **сразу показывать лендинг** (он публичный, ничего страшного).
- Параллельно продолжать асинхронную проверку сессии в фоне; если сессия найдётся позже — выполнить `navigate()` к дашборду.

### 2. `src/components/ProtectedRoute.tsx`
- Добавить таймаут **5 сек** на `getSession()`. При таймауте — редиректить на `/auth` (вместо вечного спиннера).
- Использовать `onAuthStateChange` как и сейчас — если сессия восстановится после редиректа, пользователь автоматически попадёт куда нужно.
- Добавить визуальное сообщение под спиннером после 3 сек ожидания: «Соединение с сервером...», чтобы было понятно, что не зависло.

### 3. (опционально, если останется время) `src/integrations/supabase/client.ts`
Не трогаем — файл автогенерируемый. Вместо этого все таймауты делаем на уровне вызывающего кода.

## Технические детали

```ts
// helper в src/lib/authTimeout.ts
export async function getSessionWithTimeout(ms = 2500) {
  return Promise.race([
    supabase.auth.getSession(),
    new Promise<{ data: { session: null }, error: Error }>((resolve) =>
      setTimeout(() => resolve({ data: { session: null }, error: new Error("timeout") }), ms)
    ),
  ]);
}
```

Использовать его в `Index.tsx` (2.5s) и `ProtectedRoute.tsx` (5s).

## Что это даёт
- Лендинг открывается всегда, даже если backend моргает.
- Защищённые страницы не виснут навсегда — максимум 5 секунд ожидания, потом редирект на `/auth`, где можно перелогиниться.
- Никаких изменений в логике авторизации, только защита от зависших сетевых вызовов.

## Что НЕ входит в этот план
- Не трогаем edge-функции и БД.
- Не меняем сам Supabase client.
- Не меняем поведение для уже авторизованных пользователей в нормальных условиях.
