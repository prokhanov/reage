## Причина

В `src/pages/Profile.tsx` email берётся из `supabase.auth.getUser()` (строка 95-98), а это ВСЕГДА текущая авторизованная сессия — то есть суперадмина (`prokhanov@gmail.com`). В режиме «просмотра пациента» (`viewAsUserId`) email не подменяется на email пациента, поэтому на карточке Эмиля Николаева показывается почта админа.

Данные в БД корректные — на прошлой итерации мы добавили `guard_profile_email_sync_trg`, который синхронизирует `profiles.email` с `auth.users.email`. Значит `profiles.email` для Эмиля — верный. Проблема чисто отображательная (UI).

## План устранения

### 1. Fix UI (`src/pages/Profile.tsx`)
- Расширить интерфейс `Profile` полем `email?: string`.
- Загружать email из строки `profiles` (это поле уже есть и защищено триггером).
- Использовать `profile.email` вместо `email` из auth‑сессии.
- Fallback: если `profile.email` пуст И мы НЕ в view‑as режиме — использовать `auth.getUser().email` (для старых аккаунтов без синхронизации).
- Убрать локальный state `email` — вычислять из profile.

Права/функционал суперадмина не трогаем: `useViewAsUser`, `getUserId()`, RLS, кнопки «Редактировать», `EditProfileDialog`, `PassportDataDialog`, смена пароля/телефона — всё остаётся как есть.

### 2. Проверки (защита от повторения)
- Пройти по списку страниц в патиент‑режиме и найти оставшиеся места, где данные пациента подтягиваются из `auth.getUser()` вместо `getUserId()` / профиля. Кандидаты для проверки: `Prescriptions.tsx`, `Subscription.tsx`, `SubscriptionSuccess.tsx`, `AppSidebar.tsx`. Исправляю только реальные баги отображения данных пациента, не трогая логику проверки ролей (`useUserRole`, `useSuperAdminCheck`, guards — они обязаны читать сессию админа).
- Логика чёткая: `auth.getUser()` = «кто сейчас залогинен» (роли, permissions, action‑audit). Данные о пациенте = `getUserId()` + запрос к `profiles`/`auth_admin`.

### 3. Верификация
- Открыть админом карточку Эмиля → Профиль → убедиться, что email = его настоящий (`profiles.email`).
- Выйти из view‑mode → открыть свой Профиль → email = свой.
- Проверить, что кнопки «Редактировать профиль», «Сменить пароль», «Изменить телефон» работают и не ограничены для админа в режиме просмотра.

### Что НЕ меняем
- RLS политики
- `useViewAsUser`, роли, guards
- Триггер `guard_profile_email_sync_trg`
- Разрешения суперадмина (редактирование профиля/паспорта/анкеты пациента остаётся)
