## Причина

Логин пользователя ни при чём. Ошибка происходит из-за неверной маршрутизации токена на фронте.

В `src/pages/Unsubscribe.tsx` есть регулярка `UUID_RE`, которая должна отличать токены из футера транзакционных писем (обрабатываются `handle-email-unsubscribe`) от HMAC-токенов drip-серий (обрабатываются `drip-unsubscribe`).

Проблема: токены из `email_unsubscribe_tokens` генерируются как **64 hex-символа** (32 случайных байта → hex), например `666701fae61c89901e15bb837697afa726512cf298b54d68c8f227be538eea88`. Это **не UUID-формат**. Регулярка `UUID_RE` их не распознаёт, и фронт отправляет их в `drip-unsubscribe`, где HMAC-проверка проваливается → 400 → "Edge Function returned a non-2xx status code".

Никакой привязки к логину в самих edge-функциях нет; `verify_jwt=false` уже стоит и работает (проверено curl'ом).

## Решение

Заменить неверную UUID-проверку на корректное различение:
- Drip HMAC-токен имеет формат `base64url.base64url` — всегда содержит точку.
- Queue-токен из `email_unsubscribe_tokens` — 64 hex-символа без точки.

Правило: если токен содержит `.` → drip, иначе → queue.

## Изменения

**`src/pages/Unsubscribe.tsx`**
- Убрать `UUID_RE` и переписать `detectTokenKind`: `token.includes('.') ? "drip" : "queue"`.
- Комментарии обновить, чтобы отражать реальный формат queue-токена.

## Замечание про текущую ссылку

Токен `666701fae6…` уже использован (списался при проверке функции ранее). После фикса при клике по этой конкретной ссылке отобразится корректный экран «уже отписан» / «ссылка недействительна». Новые письма будут работать штатно, независимо от того, залогинен ли пользователь и под каким аккаунтом.
