## Что делаем

Да, «шаг после регистрации с паспортом» — это шаг 3 онбординга (`/onboarding/passport`). Его убираем. Плюс переделываем баннер и добавляем отчество.

1. **Добавляем «Отчество»** в профиль и во все места, где показывается/редактируется ФИО.
2. **Расширяем баннер с паспортными данными**: пользователь проверяет и правит фамилию/имя/отчество + серию/номер в одном окне. «Заполнено» = полное ФИО + серия (4 цифры) + номер (6 цифр).
3. **Убираем шаг 3 «Паспорт» из онбординга**. Паспорт собирается только через баннер на дашборде / при записи на анализы.
4. **Проверяем и чиним логику** регистрации и записи на анализы под новые правила.

## Технические детали

- Валидация отчества: `middle_name.trim().length >= 2` (мягко, чтобы пропускать «оглы»/«кызы»). Пустое → «не заполнено».
- Нормализация: `.trim()` перед сохранением; пустая строка → `null`.
- RLS: `profiles` уже разрешает пользователю править свои поля — новых политик не нужно.
- GRANT: колонка добавляется в существующую таблицу — дополнительных GRANT не требуется.
- Типы `src/integrations/supabase/types.ts` регенерируются автоматически после миграции.
- `handle_new_user` не пишет `middle_name` — это ок, поле nullable, заполняется позже через баннер/профиль. `saveOnboardingData` тоже не трогает `middle_name`, чтобы не затирать введённое пользователем.
