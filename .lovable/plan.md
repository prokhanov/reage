# Этап 1 — расширение блока «Назначения»

## Что должно получиться у пациента в отчёте

В разделе «Назначения» отчёта будет три подблока, всегда в одном порядке:

1. **Нутрицевтики** — список добавок (как сейчас: название, форма, дозировка, схема приёма, длительность, причина, на что влияет).
2. **Питание и коррекция образа жизни** — три группы буллетов:
   - Питание (рекомендации по продуктам и режиму),
   - Физическая активность,
   - Сон и режим.
3. **Дополнительные консультации и обследования** — список «специалист → цель → триггер из анализов» (например: «Кардиолог → оценка сосудистого риска → ЛПНП 4.2 ммоль/л»).

Если у модели не получится сгенерировать какой-то подблок — он просто не появится в отчёте, остальное будет как обычно. Старые отчёты не меняются.

## Технические изменения

Все правки изолированы внутри блока «Назначения». Stage 1–4 (категорийные отчёты), резюме, биомаркеры, биовозраст, индекс здоровья — не трогаем.

### 1. База данных — без миграции

Используем уже существующую колонку `recommendations.content_json` (jsonb, nullable). Сейчас она применяется для snapshot резюме — добавим вторую запись с `type = 'Назначения'`, в которой `content_json` будет содержать структуру:

```json
{
  "lifestyle": {
    "nutrition": ["...", "..."],
    "activity":  ["...", "..."],
    "sleep":     ["...", "..."]
  },
  "follow_ups": [
    { "specialist": "кардиолог", "goal": "оценка...", "trigger": "ЛПНП 4.2" }
  ]
}
```

Поле `text` у этой записи — короткое человекочитаемое summary (для совместимости с триггером валидации `validate_recommendation_type`, который уже разрешает тип «Назначения» — нужно будет проверить, и при необходимости обновить trigger через миграцию).

Сами добавки продолжают писаться в таблицу `prescriptions` — без изменений схемы.

### 2. Edge-функция `analyze-biomarkers/index.ts`

Точечные правки в одном блоке (строки ~980–1167):

- В system-инжект (строка 1113) заменить старую инструкцию по JSON на расширенную схему:
  ```
  {
    "prescriptions": [ { name, form, dosage, how_to_take, duration,
                         prescription, reason, effect, duration_months } ],
    "lifestyle": {
      "nutrition": [ "..." ],
      "activity":  [ "..." ],
      "sleep":     [ "..." ]
    },
    "follow_ups": [ { "specialist": "...", "goal": "...", "trigger": "..." } ]
  }
  ```
- В парсере (строки 1129–1157) добавить безопасное извлечение `lifestyle` и `follow_ups` (с дефолтами и обрезкой строк по длине, как уже сделано для prescriptions).
- После того как добавки записаны в `prescriptions`, вставить **одну** запись в `recommendations` с `type = 'Назначения'` и `content_json = { lifestyle, follow_ups }` (только если хотя бы один из этих блоков не пустой).
- Резюме (`summaryPrompt`, строки 1188–1193) продолжает получать список добавок как сейчас — изменение схемы его не ломает. Опционально позже передадим туда же lifestyle/follow_ups, но не в этом этапе.

### 3. Промпты `prescriptions_system` и `prescriptions_user`

Уже содержат блоки «Питание и коррекция образа жизни» и «Дополнительные консультации и обследования». Оставляем как есть — они теперь будут реально использоваться, потому что под них появились слоты в JSON. Никаких правок в БД-таблице `ai_prompt_settings` не требуется.

### 4. Рендер отчёта на странице «Рекомендации» (`src/pages/Recommendations.tsx`)

В существующей секции с `id: 'prescriptions'` (строки ~565–570 при сборке секций и ~906–965 при отрисовке):

- После цикла отрисовки добавок добавить два подблока:
  - «Питание и коррекция образа жизни» — три подзаголовка (Питание / Физическая активность / Сон и режим), каждый — список буллетов;
  - «Дополнительные консультации и обследования» — список карточек или строк формата `специалист → цель → триггер`.
- Источник данных — найти среди `selectedReport.recommendations` запись с `type = 'Назначения'` и взять её `content_json.lifestyle` / `content_json.follow_ups`.
- Если соответствующих данных нет — подблок не отрисовывается (старые отчёты).

### 5. PDF-экспорт

В `src/lib/pdfExportHelpers.ts` (или там, где сейчас формируется секция «Назначения» для PDF) добавить ровно те же два подблока с тем же стилем, что и существующий список добавок. Источник — та же запись `recommendations` с `content_json`.

### 6. Страница «Назначения» (`src/pages/Prescriptions.tsx`)

Не трогаем. Эта страница показывает только нутрицевтики из `prescriptions` — её роль и остаётся, питание и обследования живут только в отчёте.

### 7. Админ-редактирование

В этом этапе **не делаем**. После того как новые блоки начнут стабильно генерироваться и отрисовываться, отдельной задачей добавим в `EditReportDialog` редактирование `content_json.lifestyle` и `content_json.follow_ups`.

## Что точно не меняется

- Остальные разделы отчёта (биомаркеры по системам, общее резюме, биовозраст, индекс здоровья).
- Таблица `prescriptions` и страница «Назначения» в личном кабинете.
- Все остальные edge-функции, дашборд, тренды, зоны риска.
- Существующие сохранённые отчёты (новые подблоки у них просто не отобразятся, поломки не будет).

## Риски и как они закрыты

- **Если модель вернёт пустые `lifestyle` / `follow_ups`** — запись в `recommendations` не создаётся, подблоки не рендерятся, остальной отчёт работает.
- **Если модель сломает JSON** — текущий обработчик `Failed to parse prescriptions JSON` уже есть; он пометит статус назначений как `error`, и отчёт всё равно сформируется без блока «Назначения» (как и сейчас в этом сценарии).
- **Триггер `validate_recommendation_type`** — уже разрешает тип «Назначения» (это в списке `ARRAY['Данные пациента', 'Общее резюме']` плюс категории биомаркеров). Перед записью проверим, что тип «Назначения» проходит; если нет — добавим миграцию, расширяющую разрешённые типы.

## Порядок работ

1. Проверить, проходит ли тип «Назначения» через триггер `validate_recommendation_type`; при необходимости — миграция, добавляющая его.
2. Обновить JSON-схему и парсер в `analyze-biomarkers/index.ts`, добавить запись в `recommendations` с `content_json`.
3. Расширить рендер блока «Назначения» в `Recommendations.tsx`.
4. Расширить PDF-экспорт в `pdfExportHelpers.ts`.
5. Прогнать генерацию отчёта на тестовом пациенте, проверить, что все три подблока появляются в вебе и в PDF.
