
# План: онбординг после оплаты (не связан с регистрацией)

## Итоговый поток

```
Регистрация (только аккаунт)  →  /dashboard (демо, всё доступно)
                                   ├─ гуляет, смотрит демо
                                   └─ жмёт «Оформить подписку»
                                         ↓
                                    /subscription  →  Робокасса
                                         ↓ callback
                                  /subscription/success
                                         ↓
                          /onboarding/personal → /onboarding/health → /dashboard
```

Регистрация и онбординг **полностью развязаны**. Демо доступно всем сразу после регистрации без ограничений.

## 1. Регистрация → минимальный шаг 1

- В `Register.tsx` оставляем только шаг 1 (аккаунт: имя, e-mail, телефон, пароль).
- Шаги 2 (пол/дата/вес/рост) и 3 (мед. история) — **комментируем** (не удаляем), по паттерну уже закомментированного шага оплаты.
- После успешного `signUp` + автологина → `navigate("/dashboard")`. Никаких переходов в `/subscription`.
- `handleFinalSubmit` со всей логикой записи в `weight_history`/`medical_history` — комментируем. Эта логика переезжает в онбординг.
- Draft `reage:register:*` в localStorage чистим после успеха шага 1.

## 2. Новые страницы `/onboarding/*`

Отдельный автономный флоу, не пересекается с регистрацией.

- `/onboarding/personal` — обёртка над `RegisterStep2` (пол, дата рождения, вес, рост). Заголовок «Расскажите о себе».
- `/onboarding/health` — обёртка над `RegisterStep3` (хронические, операции, лекарства, заметка). Заголовок «История здоровья».
- Финал: пишем `profiles` + `weight_history` + `medical_history` (переносим текущий `handleFinalSubmit` в новый утилитарный `saveOnboardingData(userId, data)`), ставим `onboarding_completed = true`, confetti, `navigate("/dashboard")`.

Правила:
- Прогресс «Шаг X из 2», кнопки «Назад/Далее».
- Пропуск запрещён: пройти оба шага обязательно. Значения могут быть пустыми (кроме уже валидируемых пол/дата рождения на шаге personal), кнопки «Пропустить» нет.
- Предзаполнение: при монтировании читаем `profiles` и `medical_history` — если что-то уже есть (напр., админ ввёл), поля заполнены, пользователю нужно только подтвердить.

## 3. SubscriptionSuccess → редирект в онбординг

`src/pages/SubscriptionSuccess.tsx`:
- Удаляем ветку `registerReturnStep` (устарела).
- Когда `status === "active"`: смотрим `profiles.onboarding_completed`.
  - `false` → `navigate("/onboarding/personal", { replace: true })`.
  - `true` → `navigate("/dashboard")`.
- Кнопка «Перейти в ЛК» на экране следует той же логике.

## 4. Гвард `/onboarding/*`

Внутренний гвард самих страниц онбординга:
- Требуется активная подписка (иначе `/subscription`).
- Если `onboarding_completed = true` → `/dashboard` (нельзя вернуться и переписать через URL).
- Не блокирует ни дашборд, ни другие разделы ЛК — только сам себя защищает.

**Гварда на «боевые» разделы не ставим**: пользователь может гулять по ЛК как обычно. Если закрыл онбординг — на дашборде показываем мягкий баннер «Заполните анкету, чтобы получать персональные рекомендации» со ссылкой на `/onboarding/personal`.

## 5. Приписка в разделе «Подписка»

На `/subscription` под тарифами / рядом с CTA «Оплатить»:
> «После оплаты мы попросим заполнить короткую анкету о здоровье (2 минуты) — без неё персональные рекомендации будут неполными».

## 6. База данных

Одна миграция:
- `profiles.onboarding_completed boolean not null default false`.
- Backfill: `update profiles set onboarding_completed = true` для всех текущих строк (существующие клиенты не попадают в онбординг — как договорились: «только для новых оплат»).
- Новые пользователи стартуют с `false` → онбординг активируется только после оплаты (через логику в `SubscriptionSuccess`), для нео­плативших он не показывается никогда.
- `handle_new_user()` не трогаем — поля пол/вес/рост просто перестанут приходить в metadata, и функция их уже безопасно пропускает.

## 7. Подводные камни (учтены)

1. **Двойная запись `medical_history`** — старую логику в регистрации отключаем, оставляем только в онбординге.
2. **Пользователь закрыл вкладку на онбординге** — при следующем входе гвард сам по себе не редиректит, но на дашборде висит баннер + `SubscriptionSuccess` уже отработал только один раз. Возврат — через баннер.
3. **`admin_test` оплаты** — подписка не активируется, онбординг не запускается. ОК.
4. **Режим «Просмотр как пациент»** — админ не должен попадать в чужой онбординг. `SubscriptionSuccess` в view-as режиме не показывается, гвард страницы `/onboarding/*` проверяет только собственную сессию — риска нет.
5. **`RegisterStaff`** — не трогаем.
6. **Existing draft в localStorage** — чистим при завершении шага 1 регистрации, чтобы устаревший вес/пол не всплыл.
7. **Демо-режим не ломается** — `demo_mode_enabled` живёт независимо и продолжает работать до первого реального анализа.

## 8. Порядок работ

1. Миграция: `onboarding_completed` + backfill.
2. `Register.tsx`: комментируем шаги 2/3 и `handleFinalSubmit`, после шага 1 → `/dashboard`.
3. Новые страницы `src/pages/onboarding/OnboardingPersonal.tsx`, `OnboardingHealth.tsx`, роуты в `App.tsx`, общий `saveOnboardingData` (перенос логики из старого `handleFinalSubmit`).
4. `SubscriptionSuccess.tsx` — редирект по `onboarding_completed`.
5. Приписка на `/subscription`.
6. Баннер «заполнить анкету» на дашборде, если `onboarding_completed = false` и есть активная подписка.
7. Проверочный прогон: регистрация → демо-дашборд → оплата (test mode) → онбординг → дашборд без баннера.
