
## Диагноз проблемы

По текущей логике `scoring.ts`:

1. **Порог tier 0 слишком мягкий** — `sum ≤ 10` из максимальных 34. Если пользователь везде отвечает «средне» (score 1) по 10 вопросам, он всё ещё попадает в «Похоже, вы на верном пути». Это и произошло.
2. **Tier 0 показывает только 1 домен** — остальные 5 доменов, даже если там были «1», просто исчезают. Пользователь видит один пункт (Сон) и думает: «значит, остальное в порядке».
3. **Тон «на верном пути»** активно отговаривает от сервиса — противоположность цели квиза.
4. **Маркеры даны чипами без связи** — «Ферритин / Гемоглобин / ТТГ / Глюкоза натощак» рядом с «дневной сонливостью». Пользователь не понимает, почему именно эти анализы, и как они связаны с его симптомом.
5. **Нет общей картины** — не видно, что по 5 из 6 доменов тоже есть сигналы, просто чуть слабее.

## Что меняем

### 1. Пересобрать шкалу и всегда показывать полную карту

- Пересчитать пороги tier: `≤6` → tier 0, `7–16` → tier 1, `>16` → tier 2 (плюс правило: если ≥3 вопросов с оценкой 2 — минимум tier 1; ≥5 — tier 2).
- **Всегда** рендерить компактную «Карту образа жизни» — 6 доменов с мини-баром (0/макс), отсортированные по нагрузке. Даже если по домену 0 — он виден как «спокойный». Так у пользователя формируется честная картина, а не выборка из одного пункта.
- Развернуто (с гипотезой и маркерами) показывать не «1/2/3 домена по tier», а **все домены со score ≥ 2**, минимум 2, максимум 4. Домены с score 1 сворачиваются в «Мелкие сигналы» (список без деталей). Домены со score 0 — в «Что уже работает».

### 2. Переписать headline/CTA под tier

Убрать «Похоже, вы на верном пути» полностью — даже tier 0 звучит как «база хорошая, но есть точечные сигналы, которые видно только по крови». Каждый tier получает:
- заголовок с фактом (число доменов с сигналами),
- одно предложение почему это стоит проверить,
- CTA-подсказку (не «купить подписку», а «посмотреть, что покажут ваши анализы»).

### 3. Каждый маркер — с объяснением, а не чипом

Главное изменение UX. Вместо ряда пилюль:

```text
ЧТО ЭТО ПОКАЖЕТ
[Ферритин] [Гемоглобин] [ТТГ] [Глюкоза натощак]
```

— список из 3–5 строк вида **«маркер — за какую причину отвечает»**, привязанных к конкретному ответу пользователя. Пример для «регулярная дневная сонливость»:

```text
Ферритин          Скрытый дефицит железа — самая частая
                  причина дневной сонливости у женщин 18–45.
Гемоглобин        Покажет, если сонливость уже перешла
                  в скрытую анемию.
ТТГ               Гипотиреоз даёт ровно ту же картину —
                  «сплю и всё равно устаю».
Глюкоза натощак   Скачки сахара после еды часто
                  ощущаются как «клонит в сон».
```

Технически: в `matrix.ts` вместо `markers: string[]` — `markers: { code: string; why: string }[]`. Оставляем существующие правила калибровки (пол/возраст/BMI) — они меняют состав списка, а объяснения берутся из общей мапы `MARKER_EXPLANATIONS` по (marker × domain × topQuestionId), с фолбэком на общий текст для маркера.

### 4. Добавить «Как это связано» мостик

Между «Отметили: …» и списком маркеров — одна строка типа:

> Наблюдение → возможная причина → что её проверит

Визуально: три ступеньки/чипа в ряд, чтобы читалось как логическая цепочка, а не как случайный набор анализов.

### 5. Блок «Что уже работает»

Внизу — свернутая строка «✓ По питанию, движению и привычкам сигналов нет — это хорошая база». Пользователь чувствует, что квиз его услышал, но акцент остаётся на реальных сигналах.

### 6. CTA пересобрать

- Первичный: «Посмотреть, какие анализы это покажет» → ведёт на `/prep` (запись).
- Вторичный: «Оформить годовое наблюдение» → `/register`.
Меняем приоритет: сначала мягкий шаг (анализы), затем подписка. Сейчас наоборот — и это второй фактор «мне не нужен сервис».
