## Что меняется

В таблице «Записи на анализы» (`PatientBookingsCard`) колонку «Уведомления» с двумя бейджами заменяем на кликабельную ячейку-toggle. По клику строка раскрывается вниз и показывает таблицу всех отправленных уведомлений по этой записи.

## UI раскрытой строки

Под основной строкой бронирования появляется второй `<tr>` с `colSpan` на всю ширину таблицы. Внутри — компактная таблица истории отправок, отсортированная по `created_at desc` и дедуплицированная по `message_id` (берём последний статус каждого отправления):

| Канал | Шаблон | Получатель | Статус | Кто отправил | Когда |
|-------|--------|------------|--------|--------------|-------|
| Email | analysis_booking | user@mail.ru | ✓ sent | Иван Петров | 8 дек 14:32 |
| SMS   | appointment_reminder | +7 999… | ✗ failed (текст ошибки) | Иван Петров | 8 дек 14:30 |
| Telegram | booking_status_changed | админ-чат | ✓ sent | Иван Петров | 8 дек 14:29 |

- Канал — иконка + текст (Email/SMS/Telegram).
- Статус — цветной бейдж (sent=зелёный, failed/dlq=красный, pending=жёлтый, suppressed=серый). Для failed под бейджем мелким текстом ошибка из `error_message`.
- Получатель — email или нормализованный телефон.
- Кто отправил — имя из `profiles` по `metadata.sent_by`. Если автоотправка — «Система».
- Когда — относительное время + tooltip с полной датой.

Если истории нет — строка «Уведомлений пока не было».

В свёрнутом состоянии в колонке остаётся компактный индикатор: «✓ 3» / «✗ 1» (зелёное и красное число) + шеврон `›`/`⌄`. Это сохраняет быстрый сигнал и убирает путаницу «какой именно бейдж к чему».

## Источники данных

- `email_send_log` — фильтр `metadata->>'booking_id' = bookingId`, дедуп по `message_id` (latest). Для Telegram email-функция тоже пишет туда? Нет — Telegram идёт через `telegram_notification_log`, поэтому добавляем третий источник.
- `sms_send_log` — фильтр `metadata->>'booking_id' = bookingId`, дедуп по `message_id`.
- `telegram_notification_log` — фильтр `payload->>'booking_id' = bookingId`. Если такого поля нет в существующих записях, начинаем класть `booking_id` в payload `telegram-notify` при вызове из `send-booking-telegram` (там оно уже передаётся).

Имена админов: один запрос `profiles` по списку id из `metadata.sent_by` всех трёх логов.

## Технические детали

- `useBookingNotifications` расширяется и возвращает не два «последних статуса», а массив `events: NotificationEvent[]` со всей историей. Хук используется и для компактного индикатора (считаем `sent`/`failed`), и для развёрнутой таблицы.
- `PatientBookingsCard` хранит `expandedBookingId: string | null`. Раскрытие — Tailwind-анимация `accordion-down`/`accordion-up` (есть в `tailwind.config.ts`).
- Бейдж-индикатор в колонке: маленький `Button variant="ghost"` с шевроном, цветными счётчиками и aria-expanded.
- Компонент `BookingNotificationBadges.tsx` переименовывается в `BookingNotificationsCell.tsx` (компактный индикатор) и появляется новый `BookingNotificationsHistory.tsx` (содержимое развёрнутой строки).
- Никаких изменений в edge-функциях, кроме `send-booking-telegram`: убедиться, что `booking_id` попадает в payload (для будущей фильтрации). Существующие записи без `booking_id` в payload в историю просто не попадут — это приемлемо для исторических данных.

## Что НЕ делается

- Не трогаем логику отправки и шаблоны.
- Не добавляем фильтры/поиск внутри истории — это просто хронологический список.
- Не делаем повторную отправку из истории (кнопки «Отправить напоминания» остаются в действиях записи).
